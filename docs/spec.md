# tg-monitor — спецификация v3.1

Универсальный монитор Telegram-каналов: читает поток постов из множества
источников, отбирает тематически релевантные по эмбеддингам и форвардит их
в целевые каналы. Одна тема — один целевой канал.

---

## 1. Границы v1

**Входит:** чтение источников через юзербот, эмбеддинг-отбор по нескольким
независимым темам, нативный форвард в канал темы, дедупликация, режим
калибровки порога, состояние с восстановлением после перезапуска.

**Входит:** также выбор формы публикации и вырезка фрагмента (§12) —
раньше это откладывалось, теперь это основной механизм темы.

**Не входит (v2+):** отслеживание правок и удалений в источнике, суммаризация
или переписывание постов, LLM-классификация, веб-интерфейс, аналитика
по каналам, ранжирование источников по качеству.

---

## 2. Термины

| Термин | Значение |
|---|---|
| Источник | Telegram-канал, за которым ведётся наблюдение |
| Тема | Именованный набор примеров + порог + целевой канал |
| Центроид | Усреднённый вектор примеров темы |
| Кандидат | Пост, прошедший порог хотя бы одной темы |
| Shadow-режим | Публикация с приложенной служебной строкой score для калибровки |

---

## 3. Архитектура

Пять компонентов, каждый заменяем независимо:

1. **Reader** — Telethon-клиент. Подписка `events.NewMessage` на все источники
   + периодический добор истории (интервал `catchup_interval`) на случай
   разрывов соединения. Единственный компонент, знающий про Telegram API чтения.
2. **Embedder** — обёртка над `paraphrase-multilingual-mpnet-base-v2`.
   Принимает текст, возвращает вектор. Отделён интерфейсом, чтобы модель можно
   было вынести в общий процесс (см. §10).
3. **Matcher** — считает сходство поста с центроидом каждой темы,
   возвращает список сработавших тем со score.
4. **Deduplicator** — отсекает повторную публикацию той же новости,
   пришедшей из другого источника.
5. **Publisher** — форвард в целевой канал, обработка FloodWait,
   fallback при запрете форварда.

Состояние — JSON-файл (как в digest-bot), без БД.

**Время.** Внутри системы все отметки времени — UTC: даты постов, буфер
дедупа, состояние. Локальная зона (`logging.timezone`) применяется только
при выводе — в логах и служебных сообщениях. Доменная логика (отсечка по
возрасту поста, окно дедупа) на зону отображения не завязана.

---

## 4. Конфигурация

Конфигурация разложена по **трём файлам**. Каждый перечитывается независимо
по своему mtime: правка темы не трогает аккаунт, битый `sources.yaml` не
роняет остальное.

| Файл | Что держит | Как часто правится |
|---|---|---|
| `config.yaml` | Аккаунт, служебный чат, рантайм, логирование, пути к двум остальным | Почти никогда |
| `topics.yaml` | Темы, грани, пороги, целевые каналы | Регулярно, вручную |
| `sources.yaml` | Реестр источников (§4.1) | Часто, пачками |

Пути `topics_file` и `sources_file` разрешаются **относительно каталога
`config.yaml`**, не относительно рабочего каталога процесса — под systemd
это разные вещи.

Тема — данные, не код. Правка `topics.yaml` не требует передеплоя:
файл читается по mtime, центроиды пересчитываются на лету.

**config.yaml:**

```yaml
account:
  session_path: ~/.tg-monitor/monitor.session   # вне репозитория

service_chat: "@tg_monitor_service"             # ошибки и служебные сообщения

topics_file: topics.yaml                        # темы и грани, см. §5
sources_file: sources.yaml                      # реестр источников, см. §4.1

logging:
  level: INFO                    # уровень корневого логгера
  file: logs/tg-monitor.log      # плюс stdout всегда
  timezone: Europe/Riga          # метки времени независимо от системной TZ

embedder:
  model: paraphrase-multilingual-mpnet-base-v2
  cache_dir: ~/.tg-monitor/models   # куда кладётся скачанная модель
  device: cpu
  max_chunk_chars: 400           # ограничено окном модели, см. §5.2
  min_chunk_chars: 40            # короче — приклеивается к соседнему (§5.2)

runtime:
  catchup_interval_min: 15
  dedup_window_hours: 48
  dedup_threshold: 0.85
  publish_delay_sec: 3
  max_post_age_min: 120          # не форвардить старьё после долгого простоя
  rate_limit_per_hour: null      # заготовка, в v1 выключено (см. §7)
  forward_reposts: true          # репосты внутри источников форвардим как есть
  media_group_flush_delay_sec: 2.0   # окно сборки альбома из отдельных событий
  on_forward_forbidden: copy     # copy | skip — если у источника запрет пересылки (§7)
  publish_queue_max_size: 50     # предел очереди публикации с обратным давлением
  forward_share_min: 0.6         # доля прошедших чанков, при которой форвардим целиком (§12.1)
  min_chunks_for_fragment: 3     # посты короче не режем, форвардим (§12.1)
  fragment_carry_media: false    # переносить ли медиа при вырезке (§12.3)
  fragment_on_no_link: forward   # forward | skip — если у источника нет публичного адреса (§12.3)
```

**topics.yaml** — плоский список тем, без ключа верхнего уровня
(та же форма, что у `sources.yaml` в §4.1):

```yaml
- id: topic_one
  target: "@target_channel_one"
  sources: all                 # или список id источников
  threshold: null              # null = shadow-режим, порог не задан
  soft_threshold: 0.2          # отсечка в shadow-режиме, см. §5.5
  chunk_strategy: paragraph
  facets:                      # грани темы, см. §5.1
    - id: facet_a
      examples_file: examples/facet_a.txt    # см. §4.2
    - id: facet_b
      examples_file: examples/facet_b.txt
  negatives_file: examples/negatives.txt     # опционально, см. §5.4

- id: topic_two
  target: "@target_channel_two"
  sources: [src_a, src_c]
  threshold: 0.44
  facets: [...]
```

**Правило:** `threshold: null` означает, что тема ещё не откалибрована —
она работает в shadow-режиме и публикует всё, что выше «мягкого» порога,
с приложенным score. Переход в бой = проставление числа.

### 4.1 Реестр источников

Список источников живёт отдельно от тем, в `sources.yaml`. Он пополняется
постоянно и по другой логике, чем темы: темы меняются редко и вручную,
источники добавляются пачками и отваливаются сами.

```yaml
- id: src_a
  ref: "@channel_a"
  status: active            # active | paused | unavailable
  tags: [diaspora, relocant_2022, de]
  boost: 0.03               # надбавка к score постов этого источника
  added: 2026-07-20
  note: "чем канал полезен"
```

- `status: unavailable` проставляется автоматически, если аккаунт потерял
  доступ к каналу; источник не удаляется, а помечается, и в служебный канал
  уходит уведомление.
- `tags` — свободные метки. В v1 используются только для отбора вручную
  и для `sources: [...]` в темах; машинной логики за ними нет.
- `boost` — см. §5.6. Поле необязательное, отсутствие эквивалентно `0`.

Добавление источника — два шага: аккаунт вступает в канал, затем запись
попадает в реестр. Служебная команда `/add_source` делает и то и другое,
проверяя доступ и запрет пересылки сразу.

### 4.2 Файлы примеров

Примеры граней и негативы живут **не в YAML, а в отдельных текстовых
файлах**. Причина практическая: примеры — это посты из Telegram, с
переносами строк, эмодзи и своим форматированием. В YAML такой текст
требует единообразных отступов в каждой строке, и любая вставка через
буфер обмена ломает файл. Это повторяется у каждого, кто будет настраивать
инструмент, включая конечного пользователя.

Формат файла — посты подряд, разделённые строкой из трёх дефисов:

```
Первый пример поста, как есть, с любыми отступами
и переносами строк.
---
Второй пример.
```

Правила:

- разделитель — строка, состоящая ровно из `---`;
- пустые записи между разделителями игнорируются;
- ведущие и конечные пробелы записи обрезаются;
- пути в `examples_file` и `negatives_file` разрешаются относительно
  каталога `topics.yaml`, как и остальные пути в §4;
- файлы примеров в git не попадают (это рабочие данные, не образцы),
  кроме `*.example.txt`.

Правка файла примеров подхватывается по mtime так же, как правка
конфигов, и вызывает пересчёт центроида соответствующей грани.

## 5. Отбор по теме

### 5.1 Центроиды и грани темы

Примеры — **реальные посты**, а не описания темы: описание («новости о
транспорте в Риге») живёт в другом языковом регистре, чем пост, и
систематически даёт заниженное сходство. Хранятся в отдельных текстовых
файлах, см. §4.2.

**Один пример — одна новость.** Пост-дайджест в примере смешивает
несколько сюжетов в одном векторе и уводит центроид в сторону. Если
источник пишет пачками, для примера берётся одна новость из пачки,
а не пост целиком.

Тема состоит из **граней** (`facets`), у каждой свой набор примеров и свой
центроид. Score темы = максимум по граням.

Почему не один центроид на тему: если тема широкая и объединяет разнородные
сюжеты, усреднение всех примеров в один вектор уводит центроид в
«срединную» точку, не близкую ни к одному из них. Порог тогда приходится
опускать, и вместе с целевыми постами начинает проходить всё подряд.
Грани решают это без усложнения: каждая проверяется отдельно, тема
срабатывает по лучшей.

Минимум для рабочей грани — 8–10 примеров. Меньше — центроид переобучен на
формулировки конкретных постов. Тема с одной гранью — допустимый частный
случай.

Число примеров — рекомендация, а не валидация, по тому же правилу, что
и `boost` в §5.6: грань с одним примером загружается, но пишет
предупреждение в лог. Пустая грань — ошибка конфига.

### 5.2 Чанкование

Длинный пост, усреднённый целиком, размывает тему: релевантный абзац тонет
в нерелевантных. Поэтому:

- Пост режется по абзацам (`\n\n`), короткие абзацы (< 40 символов)
  приклеиваются к соседнему.
- Score поста = **максимум** сходства по чанкам, не среднее.
- Пост целиком не усекается: длинный текст полностью разбирается
  на чанки, каждый оценивается отдельно.
- Абзац длиннее `max_chunk_chars` режется по границе предложения, если
  она есть в пределах последней трети чанка; иначе по границе слова;
  жёсткий разрез по символу — только если слово само длиннее лимита.
  Разрыв посреди слова портит токенизацию и добавляет шум в вектор.

**Окно модели.** У `paraphrase-multilingual-mpnet-base-v2` длина входа —
128 токенов, для русского текста это примерно 300–450 символов. Чанк
длиннее модель усекает **молча**: чанкер отработает верно, а хвост
потеряется уже внутри энкодера. Поэтому `max_chunk_chars` держится
в районе 400, а не подгоняется под удобство чтения.

Embedder обязан предупреждать в лог, когда токенизатор усекает вход —
иначе деградация качества остаётся невидимой. Если модель меняется,
`max_chunk_chars` пересматривается вместе с ней.

### 5.3 Что считается текстом поста

Текст + подпись к медиа. Пост без текста (чистое фото/видео/кружок)
в v1 не оценивается и не публикуется — оценивать нечего.

### 5.4 Негативные примеры

Опционально. Если заданы, считается второй центроид, и условие срабатывания:
`sim_positive - sim_negative > threshold`. Нужны для тем, у которых есть
близкий, но нежелательный сосед (условно: спорт vs. спортивные ставки).
Начинать без них, добавлять только если ложняки группируются вокруг
одного сюжета.

### 5.5 Калибровка порога

Порог не угадывается. Процедура:

1. Тема заводится с `threshold: null`.
2. Инструмент публикует в целевой канал всё, что прошло мягкий порог
   (`soft_threshold`, по умолчанию 0.2), и вторым сообщением-ответом —
   служебную строку `topic_id | score | source`.
3. Через 3–7 дней по накопленным score вручную выбирается отсечка.
4. Число проставляется в конфиг, служебные строки отключаются.

Важное следствие формата: нативный форвард нельзя редактировать и нельзя
снабдить подписью. Поэтому score идёт **отдельным сообщением-ответом**,
а не в теле поста. В боевом режиме служебные сообщения не отправляются.

`soft_threshold` живёт на уровне темы, а не глобально: разные темы
по-разному шумят на сырых эмбеддингах, и общий дефолт быстро перестаёт
подходить конкретной теме.

### 5.6 Приоритет источников (`boost`)

Иногда внутри темы одна часть материала интереснее другой, но разделять
её на самостоятельную тему с отдельным каналом не хочется. Инструмент
для этого — надбавка к score на уровне источника: `score_final = score + boost`.

Это осознанная подкрутка, а не семантика: boost не делает пост более
релевантным, он снижает планку для источника, который заведомо ближе к
интересу. Ограничения: рекомендуемые значения в пределах 0.02–0.05, boost
не применяется при калибровке порога (в логи пишутся обе величины — сырой
score и итоговый), иначе порог поедет.

Диапазон — рекомендация, а не валидация: `0` и отсутствие поля допустимы,
значение выше 0.05 принимается с предупреждением в лог. Отказ загружать
конфиг из-за boost — неверное поведение.

Второй, более честный рычаг того же назначения — **состав примеров**.
Если половина примеров грани взята из приоритетного материала, центроид
смещается туда сам, без ручных надбавок. Начинать стоит с него; boost —
добор, когда состава примеров не хватило.

## 6. Дедупликация

Одна новость приходит из нескольких каналов. Правило: публикуется первый
дошедший пост, последующие похожие глушатся.

- Хранится кольцевой буфер векторов опубликованного за
  `dedup_window_hours`.
- Новый кандидат отбрасывается, если косинус с любым из буфера
  > `dedup_threshold`.
- **Проверка и фиксация — разные операции.** Вектор попадает в буфер
  только после подтверждённой публикации, а не в момент прохождения
  фильтра. Иначе пост, который прошёл дедуп, но не был опубликован
  (запрет пересылки при `on_forward_forbidden: skip`, ошибка отправки,
  снятие по лимиту), заглушит следующий настоящий дубль той же новости —
  и она не выйдет вообще.
- Дедуп работает **в пределах одной темы**: одинаковые посты в разных темах
  — это разные целевые каналы, глушить их нельзя.
- Порог дедупа выше тематического: тут ищется почти-совпадение,
  а не тематическая близость. Стартовое значение 0.85 калибруется отдельно.

## 7. Публикация

### 7.1 Две формы публикации

У поста две возможные формы выхода, и выбор между ними делается
автоматически (§12):

| Форма | Когда | Что уходит в канал |
|---|---|---|
| **Нативный форвард** | Пост целиком по теме | Пересланный пост с атрибуцией |
| **Вырезка фрагмента** | По теме только часть поста | Новый пост с текстом фрагмента и ссылкой на оригинал |

Форвард предпочтительнее там, где он применим: он сохраняет авторство
и оформление, и не требует от нас решать, где кончается нужный кусок.
Вырезка нужна там, где нужное — одна врезка внутри длинного поста; форвард
в этом случае тащит в канал всё лишнее.

Правило выбора и механика вырезки — §12.

### 7.2 Общие правила

- **Альбомы:** пост из медиагруппы форвардится группой целиком, одним
  вызовом со списком id — иначе альбом разваливается на отдельные посты.
  Нормализованный пост хранит id всех элементов группы, а не только
  первого: без этого списка групповой форвард невозможен.
  Оценивается при этом подпись группы, а не каждый элемент.
- **Защищённый контент:** если у источника включён запрет пересылки,
  форвард падает. Fallback: копия текста + ссылка `t.me/<channel>/<id>`
  и пометка источника. Поведение управляется флагом
  `on_forward_forbidden: copy | skip`.
- **Один пост в несколько тем:** разрешено, публикуется в каждый
  сработавший целевой канал. Форма выбирается для каждой темы отдельно:
  один и тот же пост может уйти форвардом в одну тему и вырезкой в другую,
  потому что «целиком по теме» — свойство пары пост-тема, а не поста.
- **Флуд-контроль:** пауза `publish_delay_sec` между отправками,
  обработка `FloodWaitError` с ожиданием и повтором. Очередь публикации
  отделена от чтения: разбор входящего потока не блокируется ожиданием.
- **Старые посты:** при добеге истории после долгого простоя посты старше
  `max_post_age_min` не публикуются, только отмечаются как обработанные.
- **Репосты внутри источников:** форвардятся как есть, без фильтрации.
  Цепочки атрибуции считаются допустимыми.
- **Лимит публикаций на тему:** механизм заложен (`rate_limit_per_hour`),
  в v1 выключен. При включении сверхлимитные кандидаты не теряются, а
  ставятся в отложенную очередь и публикуются в следующем окне; превышение
  логируется в служебный канал.

## 8. Состояние

`state.json` — горячий файл, переписывается после каждого обработанного
поста, поэтому держится маленьким:

- `last_message_id` по каждому источнику — точка добора истории;
- версия центроида каждой темы (хэш от `examples`) — чтобы по логу было
  видно, каким набором отобран конкретный пост. Версия сохраняется
  в состоянии и сверяется при старте: расхождение с вычисленной означает,
  что грани темы правились между запусками, и это пишется в лог.
  Иначе сдвиг результатов после правки примеров выглядит необъяснимым.

`dedup-buffer.json` — буфер векторов дедупа с временными метками
и id постов. Лежит отдельно и **пишется только при изменении**, то есть
когда пост прошёл тему.

Разделение не косметическое. Буфер — это сотни векторов по 768 чисел,
единицы мегабайт; складывать его в файл, который переписывается на каждый
пост, значит гонять мегабайты сериализации там, где меняются байты.
На коротком прогоне это незаметно и проявляется через недели работы
как необъяснимая вялость.

Запись атомарная (temp + rename). Потеря файла = повторная обработка
последних постов, но дедуп не даст задвоить публикацию, если буфер уцелел;
при полной потере состояния стартуем «с текущего момента», историю не догоняем.

Отсутствие и порча `state.json` — разные события, и трактуются по-разному:

- **Файла нет** — штатный первый запуск. Уровень INFO, старт с текущего
  момента, история не догоняется.
- **Файл есть, но не читается или не проходит схему** — потеря состояния.
  Уровень ERROR с явным указанием, что `last_message_id` потеряны по всем
  источникам, плюс уведомление в служебный канал. Испорченный файл
  не затирается, а переименовывается в `state.json.bad` — иначе разбирать
  причину будет не по чему.

Отличить «файл удалили» от «файла никогда не было» изнутри процесса
невозможно, поэтому граница проходит именно здесь. Молча стартовать
с чистого листа при испорченном файле нельзя.

Для `dedup-buffer.json` правила мягче: его потеря не теряет постов,
а лишь временно снимает защиту от дублей. Отсутствие — INFO и пустой
буфер, порча — ERROR, карантин в `.bad` и пустой буфер. Ронять процесс
из-за буфера нельзя.

## 9. Ошибки и деградация

| Ситуация | Поведение |
|---|---|
| Разрыв соединения Telethon | Автореконнект, добор истории по `last_message_id` |
| FloodWait при публикации | Ожидание указанного времени, повтор, лог |
| Источник недоступен / аккаунт исключён | Лог, пропуск источника, уведомление в служебный чат |
| Форвард запрещён | Fallback по `on_forward_forbidden` |
| Ошибка эмбеддера | Пост помечается обработанным, ERROR с id и причиной — молча не терять и не ретраить бесконечно |
| Прочая ошибка при оценке поста | То же поведение, но в логе честная формулировка «ошибка при оценке поста» — причина не маскируется под ошибку эмбеддера |
| Некорректный `topics.yaml` | Старый конфиг остаётся в силе, ошибка в лог |

Ошибки и служебные сообщения уходят в отдельный служебный канал
(`service_chat`), не в целевые каналы и не в личку.

## 10. Нефункциональные требования

- Развёртывание: отдельный системный пользователь, systemd-сервис,
  по образцу digest-bot. Целевая машина — отдельный VPS под этот
  инструмент, порядка 4 ГБ памяти.
- **Модель живёт в процессе бота.** На выделенной машине запаса памяти
  хватает, отдельный эмбеддинг-сервис не нужен. Интерфейс Embedder (§3)
  оставлен абстрактным на случай, если инструмент придётся ставить рядом
  с другими потребителями той же модели — тогда вынос в общий сервис
  не затронет остальной код.
- Разработка и обкатка идут локально, на машине разработчика; перенос
  на целевой VPS — пакет 7.
- 10–30 источников: эмбеддинг каждого поста в реальном времени,
  предфильтр не нужен.
- Аккаунт юзербота: отдельный, физическая SIM, 2FA включена,
  session-файл вне git, права 600.
- Аккаунт должен быть подписан на все источники. Добавление закрытого
  канала — ручной шаг: вступить, затем внести в конфиг.

## 11. Первая тема: `expat_life`

**Формулировка:** как живётся русскоязычному человеку за пределами своей
страны — предпринимателю или просто тому, у кого получается.

Тема пережила два разворота. Сначала это были положительные новости
о бизнесе диаспоры, затем деловые новости про Россию и зарубежье без
войны, теперь — жизнь экспата. Инструмента это не касается: тема живёт
в данных, и оба предыдущих разворота обошлись без единой правки кода.

### 11.1 Почему это самая трудная из трёх формулировок

Прежние версии искали **сюжет**: деловую новость, сделку, движение рынка.
Такой сюжет — отдельный пост, и он либо есть, либо нет.

Эта версия ищет **упоминание**. Нужное рассеяно по городским и страновым
каналам крошками: в сводке недели один абзац, в подборке событий одна
строка. Отсюда два следствия, оба уже учтены:

- механизм вырезки фрагмента (§12) перестаёт быть отложенной фичей
  и становится основным способом публикации;
- состав источников меняется целиком: крупные деловые медиа под эту тему
  почти бесполезны, нужны каналы сообществ.

Есть и третья трудность, которая никуда не делась с первой версии: признак
«русскоязычный за границей» в тексте поста обычно не выражен — он следует
из того, чей это канал. Значит он снова переносится на слой источников,
а центроид отвечает только за сюжет.

### 11.2 Грани

| Грань | Что собирает |
|---|---|
| `expat_business` | Своё дело за границей: открытие, рост, сделки, перенос бизнеса |
| `expat_career` | Работа в найме, назначения, профессиональные достижения |
| `expat_community` | Сообщества, культурные и общественные проекты, инициативы |
| `expat_settling` | Обустройство: документы, жильё, школы, банки, налоги — как реально живётся |
| `local_economy` | Экономика страны присутствия: рынки, компании, регулирование, цены |

Материал прежних тем переиспользуется: `business_people` ложится
в `expat_business` и `expat_career`, а `markets_macro`, `companies_deals`
и `regulation_state` из деловой версии — прямо в `local_economy`, менять
их почти не придётся.

**`local_economy` устроена иначе остальных.** Прочие грани ищут человека:
кто-то открыл дело, кого-то назначили, кто-то обустроился. Эта ищет среду,
и признак принадлежности у неё не в тексте поста, а в том, чей это канал.
Из латвийского канала берутся экономические новости Латвии, из литовского —
Литвы. Привязка к стране получается сама собой, через подбор источников:
центроид отвечает только за «это экономическая новость», страну задаёт
источник.

Следствие: эта грань будет самой урожайной. Экономических новостей в
страновом канале много, и они пойдут потоком, тогда как истории людей
попадаются редко. Если поток окажется несоразмерным, порог выносится
на уровень грани — §11.2 спеки это допускает, это правка конфига,
не кода.

**`expat_settling` самая рискованная:** «как живётся» легко растягивается
до всего подряд. Заводится с осторожностью, первой попадает под пересмотр,
если ложняков окажется много.

### 11.3 Тональность и негативные примеры

**Фильтра по тону нет.** Тема берётся как есть: трудности, неудачи,
закрытия и убытки — такая же часть картины, как успехи. Это снимает
самое слабое место первой версии концепции: sentence-transformers плохо
различают полярность, и любой отбор по знаку давал бы шум. Теперь этого
делать не нужно.

Негативы отсекают не настроение, а **чужие темы**:

- **Военная повестка** — городские каналы пишут о ней не меньше деловых.
- **Реклама и коммерческие объявления** — класс, специфичный для этой
  темы. Каналы сообществ ими переполнены: открылся ресторан, сдаётся
  квартира, приглашаем на курсы. По языку это очень близко к нужному
  («открыл своё дело», «запустил проект»), и без явных негативных
  примеров такой контент пойдёт в канал потоком. Это главный источник
  ложняков в новой концепции.

Третий кандидат, решается по наблюдению: местная политика, не касающаяся
ни людей, ни экономики.

### 11.4 Источники

Нужны страновые и городские каналы русскоязычных сообществ. Признак
«русскоязычный за границей» должен выполняться по построению — тогда
центроид отвечает только за сюжет.

Восемь нынешних источников (РБК, «Коммерсантъ», «Ведомости», Forbes,
The Bell, «Интерфакс», «Медуза») собирались под деловую версию и под эту
тему не подходят. Исключение — каналы вроде `volna_lt`, изначально
попавшие в реестр как каналы сообщества.

**Из реестра они не удаляются, а переводятся в `status: paused`** (§4.1).
Механизм для этого уже есть: приостановленный источник остаётся в файле
со всеми тегами и заметками, но не подписывается и не читается. Работа
по их подбору не пропадает, и если заказчик снова передумает, возврат —
правка одного слова в строке.

Источники под новую тему помечаются тегом страны: он и задаёт привязку
`local_economy` к стране присутствия.

### 11.5 Что осталось собрать

Реестр страновых и городских каналов сообществ, с тегом страны у каждого.
Он же задаёт привязку `local_economy`.

Примеры по пяти граням и два набора негативов — военная повестка
и реклама. Реклама важнее: она ближе всего к нужному по языку
и без явных примеров даст основной поток ложняков.

## 12. Выбор формы публикации и вырезка фрагмента

**Задача.** Нужное часто оказывается не отдельным постом, а одной врезкой
внутри длинного поста: городская сводка недели, дайджест сообщества,
подборка новостей — и где-то в середине абзац про то, ради чего мы всё
это читаем. Форвард такого поста тащит в целевой канал одну нужную
новость и несколько посторонних.

Поэтому у публикации две формы (§7.1), и §12 описывает, как между ними
выбирается и как устроена вырезка.

### 12.1 Правило выбора: распределение оценок по чанкам

Пост уже режется на чанки, и score считается по каждому отдельно (§5.2).
Этого достаточно, чтобы решить форму публикации — ничего дополнительно
считать не надо.

Для пары «пост — тема»:

- пусть `n` — число чанков поста, `k` — число чанков, прошедших порог темы;
- если `n < min_chunks_for_fragment` — **форвард**. Пост слишком короткий,
  вырезать из двух абзацев один незачем;
- если `k / n >= forward_share_min` — **форвард**. Пост по теме целиком
  или почти целиком;
- иначе — **вырезка**. Нужное — меньшинство, остальное лишнее.

Это правило прямее, чем распознавание дайджестов по формальным признакам
(нумерация, эмодзи-разделители, взаимное сходство блоков): оно опирается
не на оформление конкретного канала, а ровно на то, что нас интересует —
какая доля поста относится к теме.

Оба параметра живут в `runtime` и калибруются на shadow-периоде.

### 12.2 Что именно вырезается

- Берётся **чанк с максимальным score**.
- Соседние с ним чанки, тоже прошедшие порог, приклеиваются к нему
  и публикуются одним фрагментом: разорванная надвое мысль читается хуже,
  чем лишний абзац.
- Прошедшие чанки, **не соседствующие** с лучшим, в v1 не публикуются —
  логируются на INFO с их score. Для платного канала один точный фрагмент
  лучше трёх россыпью; если наблюдение покажет, что так теряется важное,
  правило меняется.

### 12.3 Формат публикации фрагмента

Фрагмент выходит новым постом:

```
<текст фрагмента>

<название или @адрес источника> · <ссылка на оригинальный пост>
```

- Ссылка — `t.me/<username>/<message_id>`.
- Медиа при вырезке **не переносится**: фото и альбом относятся ко всему
  посту, а не к абзацу, и приклеенная к чужому тексту картинка вводит
  в заблуждение. Управляется флагом `fragment_carry_media`, по умолчанию
  выключено.
- Если у источника нет публичного адреса, ссылку построить нельзя.
  Поведение по флагу `fragment_on_no_link: forward | skip` — форвардить
  пост целиком либо не публиковать вовсе.

### 12.4 Дедупликация фрагментов

Дедуп (§6) работает по вектору того, что опубликовано. Для вырезки это
вектор фрагмента, а не всего поста. Это уже так и устроено: в буфер
кладётся вектор чанка, давшего максимальный score, — тот самый чанк,
который и уходит в канал.

### 12.5 Риск, который виден только на живом канале

Фрагмент, вырванный из длинного поста, может потерять контекст: «он открыл
второй цех» — кто он? В городских сводках подлежащее нередко остаётся
в предыдущем абзаце.

Заранее это не лечится: попытка автоматически определить, «самодостаточен»
ли абзац, требует понимания текста, а не сходства векторов. Смотрим
на shadow-периоде и, если случай окажется частым, приклеиваем
предшествующий чанк независимо от его score.

### 12.6 Открытый вопрос

Атрибуция при вырезке слабее, чем при форварде: подписчик видит цитату
со ссылкой, а не пересланный пост с именем канала в шапке. Для канала
с платной подпиской это стоит проговорить с заказчиком отдельно.
