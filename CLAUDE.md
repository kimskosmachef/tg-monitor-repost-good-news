# Правила работы над tg-monitor

## Роли

- **Ким** — архитектор и владелец продукта. Принимает решения, приёмку и мерж.
- **Claude (чат)** — автор спеки и промптов, ревью решений.
- **Claude Code** — исполнитель. Пишет код строго по промпту пакета.
- **Субагент-ревьюер** — проверяет готовый пакет с чистым контекстом.

## Источник истины

`docs/spec.md`. Спека выше промпта: если промпт ей противоречит или молчит —
**остановиться и спросить**, не додумывать. Claude Code не правит `docs/spec.md`;
изменения спеки идут через Кима.

## Пакеты и ветки

- Один пакет — одна ветка `feature/NN-name`, ответвляется от `main`.
- Не выходить за границы пакета. Замеченная проблема в чужой зоне —
  строкой в отчёт, а не правкой в коде.
- В отчёте по пакету: маркер полноты (закрыто / закрыто частично / не закрыто)
  по каждому пункту промпта.

## Гигиена кода

Перед каждым коммитом, без исключений:

    ruff check .
    ruff format --check .
    mypy .

Чисто — значит ноль ошибок. Не «почти». Тесты: `pytest`, юнит-покрытие
обязательно для матчера, дедупликатора и чанкования.

## Секреты

- В репозиторий не попадают: `.env`, `*.session`, реальные конфиги, `state.json`.
- В git живут только `config/*.example.yaml`.
- Права на session-файл — 600.

## Запреты

- **Молчаливые потери постов.** Каждый пост в итоге либо опубликован,
  либо помечен обработанным, либо залогирован с id и причиной. Третьего нет.
- Пустой `except`. Ловим конкретное, логируем, идём дальше осознанно.
- Хардкод id каналов, порогов, интервалов — всё из конфига.
- Установка зависимостей без записи в `requirements.txt`.

## Порядок сдачи пакета

1. Код готов, гейт линтеров и тестов пройден.
2. Отчёт с маркерами полноты.
3. Субагент-ревьюер — на входе только `docs/spec.md` и диф ветки.
4. Правки по замечаниям ревьюера.
5. Тест в тестовом канале (для пакетов с публикацией).
6. Мерж в `main`, деплой на VPS: `git pull` + `systemctl restart`.

## Коммиты

Conventional Commits, сообщение по-русски:
`feat:`, `fix:`, `refactor:`, `test:`, `docs:`, `chore:`.

## Окружение

Разработка — Linux Mint, VS Code, venv в `.venv` (torch только CPU).
Прод — Hetzner VPS, отдельный системный пользователь, systemd.
Целевая версия Python фиксируется в пакете 1 по факту версии на VPS.
